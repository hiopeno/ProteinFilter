# LightScorer 内存占用优化报告

## 1. 背景与问题描述

在真实数据训练流程中，当将训练样本条数提升至 `5000` 时，出现内存不足（OOM）或明显卡顿。该问题主要影响以下场景：

- 运行 `scripts/train_models.py` 的真实数据模式；
- 运行 `scripts/run_real_pipeline.py` 的一键全流程模式；
- 在 `matrix_size=128` 且 `train/val/test` 同时扩大时更容易触发。

该问题会导致训练流程不稳定、实验参数难以放大、结果复现成本上升。

## 2. 根因分析

定位后确认内存峰值主要来自三处叠加：

1. **特征精度默认较高**  
   距离矩阵特征默认采用 `float32`，在样本量上升时，内存增长线性且较快。

2. **构建特征时的中间拷贝峰值**  
   旧流程先将特征逐条 `append` 到 Python `list`，最后 `np.stack`；该方式在堆叠阶段会产生额外峰值内存。

3. **训练前整批转 Torch Tensor**  
   旧训练代码在进入 DataLoader 前，将完整 `x_train` 一次性转为 `torch.float32`，形成额外副本。

## 3. 优化目标

- 支持 `train=5000` 的配置更稳定运行；
- 降低数据加载与训练前的峰值内存；
- 保持现有命令使用习惯，尽量兼容旧参数；
- 让用户可按 split 精细控制样本规模，而非“一刀切”。

## 4. 已实施优化方案

### 4.1 数据层优化（`lightscorer/data.py`）

- 新增 `RealDataConfig.feature_dtype`，默认设为 `float16`；
- 新增 split 独立限额参数：
  - `max_train_samples`
  - `max_val_samples`
  - `max_test_samples`
- 特征构建流程由“`list + np.stack`”改为“**预分配数组 + 原地写入**”，减少中间峰值。

### 4.2 训练层优化（`lightscorer/train.py`）

- 新增 `_NumpyBinaryDataset`，训练数据按样本懒加载；
- 改为按 batch 将 `numpy -> torch.float32`，避免一次性复制整套训练张量；
- 预测阶段改为分 batch 推理，避免一次性将全部评估数据送入设备内存。

### 4.3 脚本参数与可用性优化

- `scripts/train_models.py` 增加参数：
  - `--max-train-samples`
  - `--max-val-samples`
  - `--max-test-samples`
  - `--feature-dtype {float16,float32}`
- `scripts/run_real_pipeline.py` 同步增加上述参数并透传至数据加载层；
- `README.md` 已补充低内存推荐命令与使用说明。

## 5. 优化效果（定性）

在当前实现下，内存占用特征如下：

- **特征内存下降**：`float16` 相比 `float32` 约减半；
- **峰值下降**：取消 `list->stack` 堆叠带来的额外峰值；
- **训练前峰值下降**：不再一次性生成整套 `torch` 张量副本；
- **参数可控性提升**：可单独放大 train、收敛 val/test，避免“总量联动放大”。

> 说明：实际节省幅度与 `matrix_size`、batch size、样本分布、系统可用内存有关。

## 6. 推荐运行配置（低内存优先）

### 6.1 仅训练（真实数据）

```bash
python scripts/train_models.py --manifest data/manifest_raw.csv --raw-lmdb-dir dataSet_withScore/casp5_to_13/data --output-dir outputs/real --model-name simple_cnn --epochs 5 --max-train-samples 5000 --max-val-samples 1200 --max-test-samples 1200 --feature-dtype float16
```

### 6.2 一键全流程（真实数据）

```bash
python scripts/run_real_pipeline.py --raw-lmdb-dir dataSet_withScore/casp5_to_13/data --output-dir outputs/real_pipeline --model-name simple_cnn --epochs 3 --max-entries 20000 --max-train-samples 5000 --max-val-samples 1200 --max-test-samples 1200 --feature-dtype float16 --recall-target 0.90
```

## 7. 兼容性与风险说明

- 兼容旧参数：`--max-samples-per-split` 仍可用；若同时设置 split 独立参数，优先使用独立参数；
- `float16` 主要用于存储降内存，训练时仍转为 `float32` 计算，通常不会显著影响稳定性；
- 若机器内存仍紧张，建议同时降低：
  - `--batch-size`
  - `--matrix-size`
  - `--max-val-samples` / `--max-test-samples`

## 8. 后续可选增强（建议）

1. 增加 `--low-memory` 一键开关，自动应用推荐参数组合；  
2. 增加运行期内存日志（每阶段打印当前/峰值内存）；  
3. 支持特征磁盘缓存（`memmap`）以进一步降低常驻内存压力。

---

**结论**：本次优化已从“数据存储精度、数据构建方式、训练加载策略、参数控制粒度”四个维度系统降低内存占用，可明显提升 `train=5000` 场景下的可运行性与稳定性。
